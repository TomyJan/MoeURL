# v0.0.1 文档工程合同化优化设计

## 背景

MoeURL 当前处于规划文档阶段。现有文档已经覆盖产品方向、功能规格、技术选型、工程计划、任务清单和验收清单，但仍存在范围冲突、规格模糊和工程合同不足的问题。

本次优化目标不是补充说明性文字，而是让文档能够稳定指导 v0.0.1 的工程开发：开发者或 AI Agent 应能根据文档明确知道要做什么、不做什么、如何实现、如何验证。

## 目标

本次优化必须达成以下目标：

1. 修正 v0.0.1 范围内的产品和规格冲突。
2. 新增统一工程合同文档，集中承载开发所需的确定性约定。
3. 明确数据库 schema、API 合同、权限初始化、Cookie Session、安全边界、事件预留和标准命令。
4. 同步任务清单和验收清单，确保每个核心能力都有可执行任务和可验证验收。
5. 保持 YAGNI，不把远期能力完整实现要求混入 v0.0.1。

## 非目标

本次不做以下事情：

1. 不实现 Go 或 Vue 代码。
2. 不创建真实数据库 migration、后端 API 或前端页面。
3. 不引入新技术栈。
4. 不扩展 v0.0.1 产品范围到统计展示、二维码、批量导入、OIDC、完整用户管理、完整域名管理等远期能力。
5. 不提交 Git commit，除非用户之后明确要求。

## 推荐方案

采用「修正文档冲突 + 新增统一工程合同文档」方案。

### 现有文档职责

- `docs/product/`：定义产品定位、版本边界和路线图。
- `docs/specs/`：定义模块级规格和长期目标，同时标注 v0.0.1 边界。
- `docs/implementation/technical-decision.md`：记录技术选型和取舍。
- `docs/implementation/technical-baseline.md`：记录工程基线和通用约定。
- `docs/implementation/v0.0.1-engineering-plan.md`：记录阶段级工程实施顺序。
- `docs/implementation/v0.0.1-tasks.md`：记录任务级执行清单。
- `docs/implementation/v0.0.1-acceptance.md`：记录验收标准。

### 新增文档职责

新增 `docs/implementation/v0.0.1-implementation-contract.md`，作为 v0.0.1 工程开发主入口。

该文档集中描述：

1. v0.0.1 范围裁决。
2. 页面与路由合同。
3. 数据库 schema 草案。
4. API 合同。
5. 权限与默认数据合同。
6. Cookie Session 安全合同。
7. 事件预留合同。
8. 标准开发、测试、构建和部署命令。
9. 产品范围、规格、任务和验收映射表。

如现有文档与工程合同存在冲突，应先修正源文档；工程合同不用于掩盖冲突。

## 关键范围裁决

### 普通用户来源

v0.0.1 必须支持普通用户登录和创建短链，因此必须提供普通用户来源。

裁决：v0.0.1 提供管理员创建普通用户的最小入口，不实现完整用户管理系统。

最小入口支持：

- 创建用户；
- 设置用户名、密码、昵称；
- 选择用户组；
- 启用或禁用用户。

不支持：

- 批量导入；
- 用户资料完整编辑；
- 用户删除；
- 多用户组；
- OIDC 绑定；
- 用户自助注册。

### 初始化权限模型

裁决：v0.0.1 不提供权限模型选择器，初始化时写入固定内置权限模型。

默认模型：

- `guest` 用户组：不可创建短链。
- `user` 用户组：可创建和管理自己的短链，可使用默认短链访问域名。
- `admin` 用户组：拥有 v0.0.1 全部权限。

### 短链详情页

裁决：v0.0.1 不实现独立短链详情页。

创建成功后提供：

- 复制短链；
- 打开短链；
- 继续创建；
- 返回我的短链列表。

短链列表承担基础查看和操作入口。完整详情页、统计详情、操作日志和模板元数据留到后续版本。

### 统计事件预留

裁决：v0.0.1 预留统一事件记录接口和事件类型，不要求事件入库、聚合、查询 API 或统计 UI。

必须预留事件：

- `short_link_opened`
- `access_condition_checked`
- `redirect_blocked`
- `redirect_initiated`
- `redirect_response_sent`

实现边界：

- 定义事件类型常量；
- 定义事件记录接口；
- 访问流程调用接口；
- v0.0.1 可以使用 no-op 实现或结构化日志。

### 短链状态

v0.0.1 必须实现：

- `active`
- `disabled`
- 软删除（通过 `deleted_at` 表达）

后续预留：

- `expired`
- `exhausted`
- `pending`
- `rejected`

### 域名策略

裁决：v0.0.1 仅使用全局默认短链访问域名。

可在模型中预留：

- 系统访问域名；
- 域名用途；
- 启用状态；
- 默认标记。

不实现：

- 用户默认域名；
- 用户组默认域名；
- 完整域名管理页面；
- 域名排序兜底策略。

## 验收标准

本次文档优化完成后，应满足：

1. 普通用户来源在产品、规格、任务和验收中一致。
2. 初始化流程不再同时出现「选择权限模型」和「不实现权限预设」的冲突。
3. 短链详情页范围唯一明确。
4. 事件预留有可验证定义。
5. 短链状态最小集明确。
6. 域名策略明确为 v0.0.1 单全局默认短链访问域名。
7. 工程合同文档包含可用于开始实现的 schema、API、权限、命令和映射表。
8. 根 README 能作为项目入口。
9. 中文排版符合项目现有规范：中英文之间留空格，中文与数字之间留空格，中文语境使用全角标点。

## 风险与控制

### 风险：工程合同与源规格重复

控制：工程合同只承载 v0.0.1 的实施确定性约定，长期目标仍放在规格文档。关键冲突必须修正源文档。

### 风险：文档改动过大导致难以审查

控制：按层次修改，先产品范围，再规格，再工程合同，再任务和验收。

### 风险：提前扩展范围

控制：所有新增内容都必须服务 v0.0.1 基础可用闭环。远期能力只允许作为字段、接口或扩展点预留，不写成必须实现流程。
