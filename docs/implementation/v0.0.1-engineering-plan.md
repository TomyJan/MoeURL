# v0.0.1 工程实施计划

## 1. 目标

本文档在 [v0.0.1 实施规划](v0.0.1-plan.md) 和 [技术基线](technical-baseline.md) 的基础上，定义可以指导开发的工程实施方案。

实现时应同时遵循 [v0.0.1 工程实施合同](./v0.0.1-implementation-contract.md)。工程合同中的 schema、API、权限、事件、标准命令和验收映射是 v0.0.1 开发的主执行入口。

v0.0.1 的工程目标是交付一个可本地运行、可构建、可测试、可 Docker 部署的 MoeURL 基础系统，完成以下闭环：

```text
首次初始化 → 本地登录 → 权限判断 → 创建短链 → 管理短链 → 访问短链直接跳转
```

## 2. 技术栈

v0.0.1 使用以下技术栈：

- 后端： Go 1.25+、Chi、SQLC、Goose、PostgreSQL。
- 前端： Vue 3、Vite、TypeScript、Vuetify 3。
- 前端包管理： pnpm。
- 状态：Pinia、TanStack Query for Vue。
- 国际化：vue-i18n。
- 表单：vee-validate、zod。
- 测试：go test、testify、testcontainers-go、Vitest、Playwright。
- 部署：Docker、Docker Compose。

详细说明见 [技术选型决策](technical-decision.md) 和 [技术基线](technical-baseline.md)。

## 3. 工程阶段

v0.0.1 工程实施分为 13 个阶段：

1. 初始化 Go 后端工程。
2. 初始化 Vue 前端工程。
3. 建立数据库迁移和 SQLC 基线。
4. 实现系统设置和初始化流程。
5. 实现用户、用户组、权限和会话基础。
6. 实现本地登录和当前用户接口。
7. 实现短链模型、创建和我的短链。
8. 实现短链访问路由和直接跳转。
9. 实现管理员创建普通用户的最小入口。
10. 实现管理员全站短链入口。
11. 实现多语言、主题和基础 UI 收尾。
12. 补齐测试、Docker 和部署文档。
13. 执行 v0.0.1 验收。


### API

```text
GET  /api/v1/init/status
POST /api/v1/init/setup
```

`GET /api/v1/init/status` 响应：

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "initialized": false
  },
  "meta": {}
}
```

`POST /api/v1/init/setup` 请求：

```json
{
  "adminUsername": "admin",
  "adminPassword": "secure-password",
  "adminNickname": "管理员",
  "siteName": "MoeURL",
  "systemDomain": "example.com",
  "shortLinkDomain": "go.example.com",
  "defaultLanguage": "zh-CN",
  "defaultTheme": "system"
}
```

### 业务规则

- 未初始化时允许访问初始化 API。
- 初始化完成后再次调用 setup 返回 HTTP `200`，响应体 `code` 为 `900101`。
- 初始化会创建内置 `guest`、`user`、`admin` 用户组、固定内置权限模型、内置 `guest` 用户、默认短链访问域名、第一个管理员和系统设置。
- 权限模型固定写入，不通过初始化请求选择。
- 初始化完成前，登录、短链创建和管理入口不可用。

### 验收

- 未初始化系统进入初始化页。
- 初始化成功后可以登录。
- 第一个用户属于 `admin` 用户组。
- 初始化不能重复执行。

### 建议提交


## 8. 阶段 5：用户、权限和会话基础

### 目标

实现当前用户识别、内置用户组、权限计算和服务端会话存储。

### 后端文件

创建或修改：

```text
internal/user/model.go

internal/user/handler.go
internal/auth/password.go
internal/auth/session.go

internal/permission/permission.go
internal/permission/service.go

internal/middleware/current_user.go
queries/user.sql
queries/permissions.sql
```

### 权限常量

```text
short_link:create
short_link:read_own
short_link:update_own
short_link:delete_own
short_link:read_all
short_link:update_all
short_link:delete_all
domain:use_default
admin:access
```

### 会话规则

- 会话 ID 通过 `moeurl_session` Cookie 保存。
- Cookie 必须设置 `HttpOnly`，`SameSite=Lax`，生产环境必须设置 `Secure`。
- 会话数据保存在 `session` 表。
- 登录成功后生成新的 session ID。
- 退出登录时删除服务端会话并清理 Cookie。
- 禁用用户不能继续执行授权操作。

### 验收

- 可以根据请求识别当前用户。
- 未登录访问者识别为 `guest`。
- 用户权限由主用户组计算。
- 无权限操作会被后端拒绝。

### 建议提交

```text
feat(权限): 添加用户组权限和会话基础
```

### API

```text
POST /api/v1/short-link/create
GET  /api/v1/short-link/list
POST /api/v1/short-link/update
POST /api/v1/short-link/delete
```

创建请求：

```json
{
  "targetUrl": "https://example.com"
}
```

创建响应：

```json
{
  "code": 0,
  "message": "OK",
  "data": {
    "shortLink": {
      "id": "...",
      "url": "https://sho.rt/abc123",
      "slug": "abc123",
      "targetUrl": "https://example.com",
      "status": "active"
    }
  },
  "meta": {}
}
```

### 业务规则

- 只有具备 `short_link:create` 的用户可以创建短链。
- v0.0.1 默认使用默认短链访问域名。
- v0.0.1 默认生成 6 位小写字母数字随机短码。
- 目标 URL 只允许 HTTP 和 HTTPS。
- 短码全系统唯一。
- 短码不能与固定路由冲突。
- 用户只能查看和操作自己的短链。
- 删除为软删除。

### 验收

- `guest` 不能创建短链。
- `user` 可以创建短链。
- 用户可以看到自己的短链列表。
- 用户不能看到他人的短链。
- 用户可以禁用、启用和软删除自己的短链。

### 建议提交


## 11. 阶段 8：短链访问和直接跳转

### 目标

实现短链访问路由，根据短码查找短链并直接跳转目标 URL。

### 后端文件

创建或修改：

```text
internal/shortlink/redirect_handler.go
internal/shortlink/redirect_service.go
internal/event/service.go
internal/http/router.go
queries/short_link.sql
```

### 路由规则

固定路由优先级：

```text
/api/v1/*
/assets/*
/
/setup
/login
/link
/admin/*
```

短码创建时必须按 [v0.0.1 工程实施合同](./v0.0.1-implementation-contract.md#33-保留路径和短码约束) 校验保留路径。

短码路由：

```text
/{slug}
```

### 业务规则

- 不存在短码返回短链不存在页或 404。
- 已删除短链按不存在处理。
- 已禁用短链展示禁用状态页。
- `active` 短链返回跳转响应。
- 访问流程调用统一事件记录接口。
- v0.0.1 可以使用 no-op 事件记录实现或结构化日志。
- 必须预留 `short_link_opened`、`access_condition_checked`、`redirect_blocked`、`redirect_initiated`、`redirect_response_sent` 事件。

### 验收

- 有效短链可以跳转。
- 禁用短链不会跳转。
- 删除短链按不存在处理。
- 固定路由不会被短码覆盖。

### 建议提交

```text
feat(跳转): 添加短链访问和直接跳转
```

## 17. 开发注意事项

- 不要在 v0.0.1 实现 OIDC、中间页、确认页、访问密码、二维码和完整统计页。
- 不要绕过权限系统直接在页面里写死能力判断。
- 不要把服务端数据长期放进 Pinia。
- 不要让短码覆盖固定路由。
- 不要硬删除短链记录。
- 不要使用 JWT 作为主登录会话。
- 不要引入 Ant Design 系 UI。

## 18. 完成标准

本工程计划完成后，应满足：

- 应用可本地运行。
- 应用可 Docker Compose 部署。
- 管理员可以初始化并登录。
- 普通用户可以创建和管理自己的短链。
- `guest` 默认不能创建短链。
- 有效短链可以直接跳转。
- 禁用和删除短链不会继续跳转。
- 管理员可以管理全站短链。
- 简体中文、英语、浅色模式、深色模式可用。
- 核心测试和验收通过。
