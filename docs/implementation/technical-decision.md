# 技术选型决策

## 1. 决策结论

MoeURL 采用以下技术栈作为 v0.0.1 及后续演进的技术基线：

```text
后端： Go 1.25+ + Chi + SQLC + Goose + PostgreSQL
前端：Vue 3 + Vite + TypeScript + Vuetify 4
前端包管理： pnpm
状态：Pinia + TanStack Query for Vue
国际化：vue-i18n
表单校验：vee-validate + zod
PWA：Web App Manifest + Service Worker
测试：go test + testify + testcontainers-go + Vitest + Playwright
部署：Docker + Docker Compose + 单 Go 服务托管前端静态产物
```

这套技术栈的目标是：稳定、现代、低魔法、易自托管、便于长期维护，并能支撑 MoeURL 后续权限、统计、多域名、跳转页模板和外部登录等扩展能力。

## 2. 选型原则

技术选型遵循以下原则：

- **稳定优先：** 优先选择成熟、社区活跃、长期维护风险低的技术。
- **现代但不猎奇：** 使用现代开发体验和工程工具，但避免小众、不稳定或维护不确定的方案。
- **自托管友好：** 部署应尽量简单，适合个人和小团队维护。
- **低魔法：** 后端数据访问、权限判断和业务流程应清晰可审查。
- **长期可演进：** v0.0.1 不实现所有远期能力，但技术基线应能承接后续扩展。
- **UI 有设计感：** 前端避免通用后台模板感，优先贴近 Material Design 3 方向。

## 3. 后端技术栈

### Go

v0.0.1 的最低 Go 版本为 Go 1.25。

Go 作为后端语言，适合 MoeURL 的 API、认证、权限、短链跳转和后台任务场景。

选择原因：

- 单二进制部署友好。
- 运行稳定，资源占用低。
- 并发能力好。
- 标准库和服务端生态成熟。
- 适合自托管产品长期维护。

### Chi

Chi 作为 HTTP 路由和中间件框架。

选择原因：

- 轻量，贴近 `net/http`。
- 中间件模型清晰。
- 适合语义化 HTTP API。
- 不强加复杂框架结构。
- 便于在项目内部保持清晰模块边界。

### PostgreSQL

PostgreSQL 作为主数据库。

选择原因：

- 关系模型成熟，适合用户、用户组、权限、域名和短链等核心数据。
- JSONB 能承载模板元数据、扩展配置和后续灵活字段。
- 统计查询能力强，适合后续访问数据分析。
- 自托管生态成熟。

### SQLC

SQLC 用于根据 SQL 生成类型安全的 Go 数据访问代码。

选择原因：

- 查询明确，没有 ORM 隐式行为。
- 生成类型安全代码。
- 复杂查询可控，适合权限和统计场景。
- 与 PostgreSQL 和 Goose 配合自然。
- 更便于审查 SQL 性能和数据边界。

### Goose

Goose 用于数据库迁移。

选择原因：

- 简单、稳定、成熟。
- 支持 SQL migration。
- 与 SQLC 的 SQL 优先方式一致。
- 迁移文件易于审查和回滚。

### 认证与会话

v0.0.1 使用 Cookie Session + 服务端会话存储。

选择原因：

- 自托管 Web 系统更适合 Cookie Session。
- 服务端可主动撤销会话。
- 用户禁用、退出登录、权限变更后更容易生效。
- 后续接入 OIDC 时可继续绑定本地用户和服务端会话。

JWT 不作为 v0.0.1 的主会话方案。

### 后端辅助库

建议使用：

- `log/slog`：结构化日志。
- `cleanenv` 或同类库：配置读取。
- `argon2id`：密码哈希。
- `testify`：测试断言。
- `testcontainers-go`：PostgreSQL 集成测试。

## 4. 前端技术栈

### Vue 3

Vue 3 作为前端框架。

选择原因：

- 适合表单、列表、设置页和管理页。
- 组件模型清晰。
- TypeScript 支持成熟。
- 与 Vuetify 4、vue-i18n、Pinia 等生态配合自然。

### Vite

Vite 作为前端构建工具。

选择原因：

- 开发启动快。
- Vue 支持成熟。
- 配置简洁。
- 现代前端项目主流选择。

### pnpm

pnpm 作为前端包管理器。

选择原因：

- 依赖安装快，磁盘占用低。
- 锁文件稳定，适合多人协作和 CI。
- 对 monorepo 和后续拆分扩展友好。
- 与 Vite、Vue 3 和 Playwright 生态配合自然。

### TypeScript

前端统一使用 TypeScript。

选择原因：

- 约束 API 响应结构。
- 降低权限、状态和表单字段错误。
- 便于长期维护。

### Vuetify 4

Vuetify 4 作为 UI 组件库。

选择原因：

- Vue 生态中成熟的 Material Design 组件库。
- 表单、导航、列表、弹窗、菜单和主题能力完整。
- 支持暗色模式和主题定制。
- 能避免 Ant Design 式通用后台风格。

使用约束：

- 不直接裸用默认主题。
- 应从前端初始化阶段建立 MoeURL 自定义主题，包括色彩、圆角、密度和暗色模式。
- 默认主题采用 Material Design 3 风格。
- 首页和短链访问页应更有产品感，避免做成传统后台页面。

### PWA

v0.0.1 从前端工程初始化阶段支持 PWA 基础能力。

选择原因：

- 自托管短链系统常被移动端重复打开，PWA 可以改善启动和安装体验。
- Web App Manifest、图标和主题色属于低成本基础设施。
- Service Worker 可先用于 App Shell 和静态资源缓存，不影响业务数据一致性。

使用约束：

- v0.0.1 不缓存登录态 API、短链业务数据和权限相关响应。
- v0.0.1 不要求完整离线业务流程。

### 状态管理

前端状态分为两类：

- **客户端状态：** 使用 Pinia，例如当前用户、权限、主题、语言和全局 UI 状态。
- **服务端状态：** 使用 TanStack Query for Vue，例如短链列表、当前用户信息和管理列表。

不要把服务端列表数据长期塞进 Pinia。

### 表单和校验

使用 vee-validate + zod。

职责划分：

- zod 定义前端表单校验规则和类型。
- vee-validate 负责表单状态、错误展示和提交流程。
- 后端仍必须执行权威校验，前端校验只用于体验。

### 国际化

使用 vue-i18n。

v0.0.1 至少支持：

- 简体中文。
- 英语。

## 5. 测试技术栈

### 后端测试

后端测试使用：

- `go test`：标准测试入口。
- `testify`：断言和测试辅助。
- `testcontainers-go`：需要真实 PostgreSQL 的集成测试。

测试重点：

- URL 校验。
- 短码生成和保留路径校验。
- 权限判断。
- 初始化流程。
- 登录和会话。
- 短链创建、禁用、软删除和访问。

### 前端测试

前端测试使用：

- Vitest：单元测试。
- Vue Testing Library：组件测试。
- Playwright：端到端测试。

测试重点：

- 首页创建区域权限状态。
- 登录流程。
- 我的短链列表操作。
- 管理员全站短链入口。
- 多语言切换。
- 主题切换。

## 6. 部署技术栈

v0.0.1 推荐部署形态：

```text
浏览器
  ↓
Go 服务
  ├─ /api/v1/*       后端 API
  ├─ /assets/*       前端静态资源
  ├─ 固定前端路由     SPA fallback
  └─ /:slug          短链跳转
  ↓
PostgreSQL
```

生产环境优先提供 Docker 和 Docker Compose。

推荐方式：

- 前端使用 Vite 构建为静态资源。
- Go 服务在生产环境托管前端静态产物。
- API、前端页面和短链跳转由同一个 Go 服务处理。
- PostgreSQL 作为独立服务运行。

这种方式适合自托管用户，部署简单，也保留未来前后端分离部署的可能。

## 7. 不采用的方案

### 不采用 Gin 作为首选路由框架

Gin 成熟且流行，但 MoeURL 不需要较重的框架式结构。Chi 更轻量，也更贴近 Go 标准库风格。

### 不采用 GORM 作为首选数据访问方案

GORM 使用广泛，但隐式行为较多。MoeURL 的权限、短链、统计和审计查询需要清晰 SQL，SQLC 更可控。

### 不采用 JWT 作为主会话方案

JWT 适合开放 API 和多服务认证场景。MoeURL v0.0.1 是自托管 Web 系统，Cookie Session 更适合用户禁用、登出、权限变更和 OIDC 后续接入。

### 不采用 Ant Design 系 UI

MoeURL 需要更现代、更有产品感的界面。Vuetify 4 更贴近 Material Design 3 方向，符合项目视觉偏好。

### 不优先采用 SQLite

SQLite 对个人部署友好，但 MoeURL 后续会涉及访问事件、统计聚合、多用户管理和后台查询。PostgreSQL 更适合作为长期主数据库。
