# v0.0.2 实施计划

## 1. 目标

v0.0.2 是 v0.0.1 合并后的第一轮质量补强版本，目标是把基础可用闭环推进到更适合自托管试用和后续功能演进的状态。

本版本的主线是：

```text
验收记录闭环 → 管理员用户管理 → 列表筛选体验 → 测试和部署验证补强
```

v0.0.2 不实现 v0.1.0 的统计、中间页跳转、访问密码、二维码和短链过期时间。

## 3. 阶段划分

v0.0.2 分为 5 个阶段：

1. 文档和验收记录同步。
2. 管理员用户管理最小闭环。
3. 短链列表筛选和搜索体验。
4. 测试、E2E 和覆盖率门禁补强。
5. 部署文档和发布前验证。

每个阶段都应产出可验证结果，并尽量保持独立提交。

## 4. 阶段 1：文档和验收记录同步

### 目标

让项目文档准确反映 v0.0.1 已完成、v0.0.2 成为当前优先目标。

### 实施内容

- 更新文档中心阅读顺序。
- 更新产品总览的当前优先目标。
- 更新路线图，插入 v0.0.2 质量补强闭环。
- 更新实施文档索引。
- 在 v0.0.1 验收清单中补充已合并后的状态说明。

### 验收

- v0.0.1 文档保留为历史和兼容性依据。
- 文档没有把 v0.1.0 能力误标为 v0.0.2 必须实现。

## 5. 阶段 2：管理员用户管理最小闭环

### 目标

在 v0.0.1 管理员创建用户入口之上，补齐用户列表和最小维护能力。

### 后端建议 API

```text
GET  /api/v1/admin/user/list
POST /api/v1/admin/user/update
POST /api/v1/admin/user/reset-password
```

### 业务规则

- 所有接口需要 `admin:access`。
- 列表展示账号、昵称、用户组、状态、是否内置、创建时间和更新时间。
- `update` 只允许修改普通用户的昵称和状态。
- `reset-password` 只允许重置普通用户密码。
- 内置 `guest` 用户不可修改、不可禁用、不可重置密码。
- 管理员不能通过本阶段能力删除用户。
- 禁用用户后，该用户不能登录；如果已有会话，后续授权操作必须失败。

### 前端页面

```text
/admin/users
```

页面应支持：

- 用户列表。
- 状态标识。
- 启用或禁用操作。
- 修改昵称。
- 重置密码。
- 空状态、加载状态和错误状态。

### 验收

- 管理员可以完成普通用户的查看、禁用、启用、昵称修改和密码重置。
- 普通用户无法进入用户管理页，也无法调用用户管理 API。
- 内置 `guest` 用户不会被管理操作修改。

## 6. 阶段 3：短链列表筛选和搜索体验

### 目标

提升 v0.0.1 已有短链列表在真实使用中的可扫描性。

### 后端建议

扩展既有列表接口参数：

```text
GET /api/v1/short-link/list?page=1&pageSize=20&status=active
GET /api/v1/admin/short-link/list?page=1&pageSize=20&status=disabled&q=alice
```

### 业务规则

- `status` 只接受 `active`、`disabled` 或空值。
- `q` 用于管理员列表中的短码、目标 URL、用户名或昵称模糊搜索。
- 普通用户列表不得通过搜索泄露他人短链。
- 筛选和搜索只作用于未软删除短链。

### 前端要求

- 我的短链页支持状态筛选。
- 管理员全站短链页支持状态筛选和关键词搜索。
- 筛选条件变化后刷新 TanStack Query 查询。
- 空结果应显示清晰提示。

### 验收

- 用户可以筛选自己的 active 或 disabled 短链。
- 管理员可以按状态和关键词查找全站短链。
- 普通用户仍不能查看他人短链。

## 7. 阶段 4：测试、E2E 和覆盖率门禁补强

### 目标

让 v0.0.2 新增能力具备可回归验证，并保持现有 100% 覆盖率门禁。

### 后端测试

至少覆盖：

- 管理员用户列表权限判断。
- 普通用户访问用户管理 API 被拒绝。
- 内置 `guest` 用户不可修改。
- 用户禁用后无法登录或继续执行授权操作。
- 用户列表更新和密码重置成功路径。
- 短链列表状态筛选。
- 管理员短链关键词搜索。

### 前端测试

至少覆盖：

- 用户管理页权限状态。
- 用户状态切换。
- 重置密码表单。
- 我的短链状态筛选。
- 管理员短链搜索和空结果。

### E2E 测试

至少覆盖：

```text
初始化 → 登录管理员 → 创建普通用户 → 禁用普通用户 → 普通用户登录失败
初始化 → 登录管理员 → 创建短链 → 管理列表按状态筛选 → 搜索短链
```

### 验收

- 后端覆盖率门禁保持 100%。
- 前端覆盖率门禁保持 100%。
- E2E 覆盖 v0.0.2 新增主流程。

## 8. 阶段 5：部署文档和发布前验证

### 目标

补齐自托管部署者执行 v0.0.2 验证所需的明确步骤。

### 实施内容

- README 补充完整验证命令。
- README 补充 Docker Compose 健康检查和初始化入口验证。
- 如新增环境变量，补充说明并更新技术基线。
- 记录已知耗时命令，例如 Docker 首次构建。

### 验收

- 新开发者可以按 README 完成本地质量检查。
- Docker Compose 启动后可以访问 `/api/v1/health`。
- Docker Compose 启动后可以进入初始化入口。

## 9. 发布前必跑命令

后端：

```bash
go test ./...
go vet ./...
go test ./internal/auth ./internal/db ./internal/http ./internal/permission ./internal/shortlink ./internal/system ./internal/user -coverprofile="$PWD/coverage.out"
node scripts/go-coverage-threshold.mjs "$PWD/coverage.out" 100 --include-from=scripts/go-coverage-targets.txt --exclude-blocks-from=scripts/go-coverage-excluded-blocks.txt
```

前端：

```bash
cd web
pnpm lint
pnpm typecheck
pnpm test
pnpm test:coverage
pnpm build
pnpm test:e2e
```

部署：

```bash
docker compose up --build
```

涉及 Docker 路径时，必须等待命令明确退出或完成健康检查后再记录结果。
